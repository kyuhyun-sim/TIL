readme.md, issue, pr, project, milestone, wiki

readme.md는 언제 작성할지 정해진건 아니나 몰아서 하는 것 보다 중간중간 작성하는 것이 좋다.
issue 발행은 버그였을때나 새로운 기능을 추가했을때도 하고, 모르는거 팀원에게 도와달라고 도움을 요청하거나, 버그 고쳐야할거 같다 의견 제시할때도 사용한다.
issue로 버그를 고쳐야겠다, 고쳐달라 한다음에 pr을 통해 고친다.

readme.md : 우리프로젝트는 이런거다 + 트러블슈팅 -> pr : 깃에 팀 작업물을 올리기 전에 확인과 피드백(코드리뷰)을 받을 수 있음 -> milestone : 팀이 해야할 작업 + 작업 진행정도

깃허브 플로우랑 깃 플로우랑 다르다.
깃 플로우는 안드로이드 같은 좀더 복잡한 프로젝트에 사용한다.


지난번에 했던 깃액션을 사용하여 스프링 서버를 자동 배포하는 기능을 마저 구현해보았다.
빌드까지는 문제없이 끝낼수 있었으나, S3에 빌드된 파일을 업로드 하는 부분에서 오류가 발생했다.
문제의 원인은 총 3가지 였다.
먼저 S3 버킷을 생성하지 않았기에 버킷 네임을 찾지 못한거였다.
두번째로는 s3의 region 을 시크릿에서 가져오지 못하는 문제가 발생하였다.
region의 값을 경우 한국을 표시하는 ap-northeast-2 였기에 굳이 시크릿 경로에 넣을 필요가 없다고 판단하여 'aws-region: ap-northeast-2' 이처럼 직접 값을 넣어주었다.
마지막 문제는 보안 토큰키가 유효하지 않다는 에러 메세지가 출력되는 것이었다.
원인은 aws의 액세스 키와 시크릿키를 시크릿 경로에서 가져오지 못하는 문제가 발생하였기 때문이었다.
직접 경로를 재정의 해줌으로서 문제를 해결할 수 있었다.
